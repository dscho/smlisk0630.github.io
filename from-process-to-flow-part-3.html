From Process to Flow: Part 3

In Part 2, we outlined our Flow project, installed packages, and started building the Flow. We created variables. Now it’s time to build our first screen, the screen where users will enter their information!

Create Elements

We’re not finished creating variables, but for now we’ll turn our attention to creating the elements for our Flow.

Element 1: New Case Screen

First things first: we need to create the first screen users will see when starting your Flow.

  1. In the Toolbox, click “Elements.”
  2. You should see an element at the top of the menu that says “Screen” with a bright blue icon of a computer screen beside it. Click the icon and drag it onto the canvas on the right.
  3. On the right, enter a Name and Description for this screen, e.g. “New Case” (“New_Case” for the API Name).
  4. In the “Configure Frame” menu, uncheck the “Show Header” box.
  5. Click the arrow to the left of “Control Navigation” to show the options available for moving through the Flow. Uncheck “Previous” and “Pause,” but leave “Next or Finish” checked.
Screen Components: Type
  1. Now it’s time to start adding Screen Components, which can be dragged and dropped from the menus on the left.
  2. Scroll down to the bottom to “Custom.” If you have installed the packages in FlowBasepacks, you will see an option labelled “Quick Choice.” Click it and drag it onto the canvas in the middle.
  3. Enter “Type” as the API name, since this field will map to the Type field on the Case object.
  4. Display the choices as “Picklist” and select “A Picklist field” for the datasource.
  5. Select “Case” as the Object and “Case Type” as the field.
  6. In the “Filter on Record Type ID” box, click in the box and scroll down until you see your formula variable. Click it.
  7. Select the “Required” box and enter the label you want to appear for this field on the screen. I entered “Type of Case” to make it clear to users what the options in the field referred to.
Screen Components: Lookup Fields

If you want to relate a Case to another record, such as an Account or Opportunity, you will add a Lookup component here.

  1. Choose an API Name for the component and enter it in the “API Name” box.
  2. Find the API Name for the existing lookup field on the Case object and enter it here. For example, the API name for the Account lookup field on my Case object is “AccountId.”
  3. Enter the label you want users to see for this field, such as “Name of Account.”
  4. Enter “Case” in the “Object API Name” box.
  5. In the “Record ID” box, choose “New Resource” to create a new variable that will store the ID of the related Account record.
    1. Select “Variable” as the Resource Type.
    2. Enter an API Name and Description.
    3. Select “Text” as the Data Type.
    4. In the “Default Value” box, type “Account” and select the “Accounts” option under Screen Components. It may say “flowruntime:lookup” under the word “Accounts”.
    5. Click “recordId” to store the ID of the Account record the user is entering.
    6. Choose “available for output,” then click “Done.”
    7. Click “Set Component Visibility” to display the options for making this field visible. These conditions may remind you of Report Filters or Formulas in which you can choose “AND” or “OR” conditions.
    8. Feel free to play around with the options if you’d like. In my Flow, I chose “Any Condition Is Met (OR).” Next you will add your Condition(s).
    9. In my Flow, I wanted the field to appear based on which Type the User chose for the Case. I wanted it to appear if the user chose either “Problem” OR “Question” for the Case Type.
      1. In the “Resource” box, enter “Type” and choose the “Type” option under “Screen Components.” “c:fsc_quickChoiceFSC” may appear under the word “Type.”
      2. Scroll down in the menu that appears and click “Value.” Your Operator should be “Equals.” Now your Condition states that visibility of the lookup field will depend on the value of the Case Type field.
      3. Wait! You’re not done! You still have to specify which value(s) visibility should depend on. This part’s easy—just enter the value, e.g. “Problem” or “Question.”
      4. Repeat the above 3 steps for all Conditions you want to create.
Screen Components: Priority

If you want your users to have the option to tell you how urgent their Case is, use the Priority field on the Case object. Follow the steps above in the “Screen Components: Type” section to do this, but choose a different Master Label to be visible to users.

Screen Components: Subject and Description

You’ll want users to be able to tell you what the Case is about and provide details. They can do so via the Subject and Description fields on the Case object.

For the Subject field:

  1. Drag the Text Screen Component onto the canvas.
  2. Enter the label and “Subject” for the API Name. Check the “Require” box.
  3. Repeat these steps for the Description field, but instead of a Text Screen Component, use the Long Text Area Component.
Screen Component: Attachments Checkbox

Now we’re going to add a checkbox so users can indicate whether they have any files, such as screenshots, they want to include with the Case. Note: Because Flow needs to create the Case record first, users cannot upload the attachments until the second screen.

  1. Drag the Checkbox Screen Component onto the canvas.
  2. Choose the label. I chose “Need to Upload Attachments?” since it was a clear indicator of the purpose of the box.
Screen Component: User Lookup Field

We’ve already covered lookup fields, but the User lookup field is especially important. When set up correctly, this field will auto-fill with the name of the User submitting the Case. If you haven’t already set up a User lookup field on the Case object, you can follow the instructions here.

Once that’s done, we put that field in the Screen Component.

  1. Drag the Lookup Component onto the canvas.
  2. Enter “User” for the Component API Name, then the API Name of the User lookup field on the Case object (which is likely to be “User__c”) in the Field API Name box.
  3. Enter a Label the user will understand, e.g. “Your Name,” then “Case” for the Object API Name.
  4. For the record Id, we’re once again going to create a new variable.
    1. Click “New Resource” and select “Variable” as the Resource Type.
    2. Enter an API Name, e.g. “varUserId,” and Description.
    3. Select “Text” as the Data Type.
    4. In the “Default Value” box, type “User” and select the “$User” option under “Global Variables.”
    5. Enter “ID” and choose “Id” to store the ID of the User record. Your default value will look like “{!$User.Id}."
    6. Choose “available for output,” then click “Done.”
Screen Component: Record Type Developer Name

Way back at the beginning of all this, you created 2 Record Variables: a Data Variable and a Record Type Variable. Now you’re going to use the Record Type Variable to ensure the Case will have the correct Record Type.

  1. Drag a “Display Text” Component onto the Canvas. Don’t worry, despite the name of the Component, this won’t be visible to users.
  2. Enter an API name, e.g. “Record_Type.”
  3. In the “Insert a resource…” box, enter the name of the Record Type Variable you created, then click on it.
  4. Scroll down to the “DeveloperName” option and click on it. Click “Done.”
  5. Click “Save.”

Final Step

Don’t forget to connect your Start element to the screen element!
